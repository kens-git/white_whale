Intro

Welcome to the first installment in the 'White Whale' social network development blog! This blog documents the progress I make on the project and will include my commentary as I explain the decisions I make: with the goal being to provide new developers with exposure to a full stack web application in development. Hopefully, this will help bridge the gap for new developers between developing small demo projects and developing a reasonably production-ready project from a given specification.

This blog will also demonstrate how to handle new features and changes to existing features. I'm sure I'll get new ideas that I want to add or may even miss a detail in the implementation of a feature and have to modify it later.

The project will use Django with Python for the back end and React with TypeScript for the front end, though my intention is to explain my reasoning so that it will be applicable to any frameworks.

The focus for the forseeable future will be on the back end. The back end is the gatekeeper of the application, so it's important to get it right first before thinking about the front end.

This blog is solely focused on development, and I won't be discussing:

  • Specific development setups (IDE, plugins)
  • Coding style
  • Specific toolchains and other code quality tools
  • Deployment (since there's too many possibilities)

The code will be put into the following repo: https://github.com/kens-git/white_whale. Note that the Docker files, scripts, and other configuration files are in a state of flux and are subject to change at any time, so don't rely on them. The app/config and app/core folders are where the code from this blog will be.

The Project

As stated, the project is a social network with the features expected in a modern social network:

  • User Profiles
  • Media Albums
  • Posts
  • Comments
  • User Feed/Recent Activity (both user-specific and site-wide)
  • Friends
  • Messages
  • Chat
  • Groups
  • Buy/Sell
  • Blogs
  • Job Board
  • Search
  • All Users Page
  • Admin Panel
  • Feature Request Page

This feature set is pretty ambitious, so the initial MVP will only include:

  • User Profiles:
    • Name (first, middle, last)
    • Location (city, state/province, country)
    • Personal about message
    • Status
    • Birthday
    • Email
    • Website
    • Phone Number
  • Media Albums:
    • Users can upload files into albums.
    • Albums have an associated name and description.
    • Users can comment on uploaded files.
    • Albums will show when they last had files added.
  • Posts
  • Comments
  • User Feed

    The user feed displays site activity that's relevant to the user, such as comments on their content or replies to their posts.

  • Recent Activity

    Recent activity is the same as the user feed, but contains activity from all users.

  • Messages
  • Groups

    Groups can be created for events, special interests, etc.

  • All Users Page

    A basic list of all users on the site.

  • Admin Panel
    • Admins can add new users.
    • Admins can edit and delete content, and the offending user should be notified.
  • Feature Request Page

    Gives an input form where users can request features they want added.

Assumptions

The project will be developed with a few assumptions in mind:

  • The expected number of registered users will range from a few dozen, up to a few thousand.
  • The maximum number of active users will be between 20 to 100.
  • Per the previous two points, optimizations aren't a priority.
  • One or more users act as administrators and will control registrations. This means new users will only be created by an admin.
  • Trust between users is assumed to be high, which allows the implementation to be simpler. For instance, there's no need to hide the content of one user from another via privacy settings: it's assumed every user knows the other users and willingly shares their content with them.
  • While user content is available to all other users, nothing is visible to anyone without an account ('walled garden').

While a site like this won't earn us trillion dollar valuations anytime soon, the simplicities these assumptions bring mean we can focus on adding features first and optimizing later: if we even have to at all.
